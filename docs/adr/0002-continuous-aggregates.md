# ADR-0002 — Continuous aggregates для даунсемплинга

**Статус:** принято · **Дата:** 2025

## Контекст

График на фронте показывает 5m/1h свечи, а скользящие метрики считаются на 5m.
Хранить отдельно мы умеем только 1m (сырьё). Пересчитывать 5m/1h на каждый запрос
из миллионов 1m-строк — дорого и медленно.

## Решение

Завести **continuous aggregates** `ohlcv_5m` и `ohlcv_1h` поверх гипертаблицы `ohlcv`
с политиками инкрементального обновления.

## Почему

- Агрегаты пересчитывают только «хвост» (новые бакеты), а не весь датасет.
- OHLC агрегируется корректно: `first(open)`, `max(high)`, `min(low)`, `last(close)`,
  `sum(volume)` — через `time_bucket`.
- Фронт и аналитика читают готовый компактный агрегат, а не сырые 1m.

## Альтернативы

- *Считать на лету оконными функциями по 1m* — корректно, но тяжело на каждый запрос.
- *Materialized view + REFRESH по cron* — полный пересчёт, не инкрементальный.

## Последствия

- Между приходом 1m и появлением в 5m есть лаг (политика `end_offset`) — приемлемо,
  свежую формирующуюся свечу всё равно даёт live-стрим через WS.
- Нужно держать retention 1m достаточным, чтобы агрегаты успевали посчитаться.
