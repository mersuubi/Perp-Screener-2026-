# ADR-0003 — Redis pub/sub между ингестом и API

**Статус:** принято · **Дата:** 2025

## Контекст

Live-тики приходят в ингест-сервис (он держит WS Binance). Их надо доставлять
браузерам через WebSocket API. Ингест и API — разные процессы (см. ADR-0004).

## Решение

Ингест **публикует** свежие тики в каналы Redis (`ticks:kline`, `ticks:mark`); API
**подписан** на них и фанаутит подключённым WS-клиентам.

## Почему

- **Развязка**: ингест ничего не знает про клиентов и их количество; API ничего не
  знает про биржу. Каждый делает одно дело.
- Несколько инстансов API могут подписаться на один канал — горизонтальное
  масштабирование фанаута без изменений в ингесте.
- Redis уже удобен как лёгкий кэш/шина; pub/sub — встроенный примитив.

## Альтернативы

- *Прямой проброс из ингеста в API по сокету* — жёсткая связность, ручной реконнект.
- *Слушать WS Binance прямо из API* — дубль коннектов, API падает вместе со стримом,
  теряется граница ответственности.
- *Kafka* — оверкилл для пет-проекта; добавляет тяжёлую инфраструктуру.

## Последствия

- Redis pub/sub — at-most-once (если нет подписчика, сообщение теряется). Для live-UI
  это нормально: история всё равно лежит в БД, потеря одного тика некритична.
