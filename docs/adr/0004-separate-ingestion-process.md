# ADR-0004 — Отдельный ингест-процесс

**Статус:** принято · **Дата:** 2025

## Контекст

Сбор данных (WS-коннекты, бэкофилл, запись в БД) и обслуживание клиентов (REST/WS API)
— разные по характеру нагрузки задачи. Возникает соблазн сделать «всё в одном» процессе.

## Решение

Выделить **ingestion** в самостоятельный сервис/контейнер, независимый от API.

## Почему

- **Изоляция отказов**: падение/перезапуск API не разрывает WS-стрим и не теряет
  данные; падение ингеста не делает API недоступным (он продолжает отдавать историю).
- **Разный жизненный цикл**: API деплоится/масштабируется часто и независимо; ингест —
  долгоживущий stateful-воркер с одним WS-коннектом на биржу.
- **Чистая граница ответственности** — ровно то, что ждёт интервьюер: «почему не
  монолит?» → потому что у процессов разные SLA и причины перезапуска.

## Альтернативы

- *Монолит (ингест + API в одном процессе)* — проще на старте, но связывает судьбы
  двух разных нагрузок; рестарт API ради нового релиза рвёт стрим.

## Последствия

- Нужна шина между процессами для live (см. ADR-0003).
- Чуть больше движущихся частей в docker-compose — оправдано целями проекта.
