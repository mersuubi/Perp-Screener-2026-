# ADR-0001 — TimescaleDB вместо голого PostgreSQL

**Статус:** принято · **Дата:** 2025

## Контекст

Основная нагрузка — тайм-серии: 1m OHLCV по 40+ инструментам (десятки тысяч строк
в сутки), плюс выборки по диапазону времени и даунсемплинг для графиков. Нужно
эффективно писать, хранить историю и быстро читать «последние N свечей по символу».

## Решение

Использовать **TimescaleDB** — расширение PostgreSQL для тайм-серий.

## Почему

- **Гипертаблицы** автоматически партиционируют данные по времени (chunks). Это даёт
  быстрые выборки по диапазону и дешёвое удаление/сжатие старых чанков — без ручного
  партиционирования.
- **Continuous aggregates** считают 5m/1h инкрементально из 1m. Альтернатива на голом
  Postgres — materialized view + ручной `REFRESH` (полный пересчёт, дорого) или
  триггеры (сложно и хрупко).
- **Политики** сжатия и retention — декларативные фоновые джобы, не cron-самопал.
- Это **по-прежнему PostgreSQL**: SQL, asyncpg, экосистема, индексы — всё работает.

## Альтернативы

- *Голый PostgreSQL* — пришлось бы руками делать партиционирование и даунсемплинг.
- *ClickHouse/InfluxDB* — мощно для тайм-серий, но добавляет отдельную систему и
  свой диалект; для масштаба пет-проекта это оверкилл, а Postgres-навык ценнее.

## Последствия

- Зависимость от расширения (образ `timescale/timescaledb`).
- Нужно понимать chunk interval и политики агрегатов — задокументировано в SQL.
