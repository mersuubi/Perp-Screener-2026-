# PRD — Perp Screener

> Product Requirements Document. Короткий, по делу. Версия 1.0.

## 1. Проблема

Трейдер деривативами держит в фокусе десятки бессрочных фьючерсов (perpetual
futures). Чтобы вовремя замечать аномалии — всплеск объёма, разгон волатильности,
перегрев funding rate, резкий приток/отток открытого интереса — нужно постоянно
мониторить рынок по нескольким условиям сразу. Руками по 40+ инструментам это
невозможно: внимание не масштабируется.

## 2. Целевой пользователь

- **Активный трейдер перпов** — ищет инструменты «на движении» под intraday/swing.
- **Аналитик рынка** — хочет воспроизводимые срезы: «покажи, что было перегрето в момент X».

## 3. Цель продукта

Автоматизировать скрининг: непрерывно собирать рыночные данные, считать набор
метрик и отдавать **отсортированную витрину** + давать сохранять прогоны скрининга
как воспроизводимые артефакты.

## 4. Функциональные требования

| # | Требование | Приоритет |
|---|---|---|
| FR-1 | Ингест 1m OHLCV, funding rate, open interest по 30–50 перпам с Binance | MUST |
| FR-2 | Идемпотентное хранение тайм-серий, даунсемплинг 1m→5m→1h | MUST |
| FR-3 | Метрики: доходность 5м/1ч/24ч, z-score объёма, волатильность, экстремум funding, ΔOI | MUST |
| FR-4 | Рыночные ранги/перцентили по инструментам (объём, волатильность, funding) | MUST |
| FR-5 | Запуск скринера по набору фильтров + сохранение прогона (run + hits) | MUST |
| FR-6 | REST API на историю/метрики, WebSocket на live | MUST |
| FR-7 | Фронт: таблица метрик (сортировка), сетка live-графиков, панель фильтров | MUST |
| FR-8 | История прогонов скринера (просмотр прошлых запусков) | SHOULD |
| FR-9 | Корреляционная матрица, снапшот-хитмап стакана, бэктест-модуль | COULD (стретч) |

## 5. Нефункциональные требования

| # | Требование |
|---|---|
| NFR-1 | Все времена — UTC, `timestamptz`. Никаких локальных часовых поясов в тайм-серии. |
| NFR-2 | Ингест устойчив к обрывам WS: авто-реконнект + докачка пропущенного интервала. |
| NFR-3 | Переподключение/повторный бэкофилл не создают дублей (идемпотентность по ключу). |
| NFR-4 | Аналитика считается в БД (SQL), не в приложении. |
| NFR-5 | Соблюдение rate limits Binance (пагинация пачками с паузами). |
| NFR-6 | Деплой воспроизводим (`docker compose up`), демо — read-only. |
| NFR-7 | Сервисы независимы: падение ингеста не роняет API. |

## 6. Метрики успеха

- БД наполняется историей за N дней по всем инструментам без дублей.
- Витрина метрик обновляется ≤ 30с; live-свеча на графике — в реальном времени.
- Прогон скринера возвращает корректный набор и сохраняется (виден в `/screener/runs`).
- Рекрутёр за 30с README видит: диаграмму, живое демо, нетривиальный SQL.

## 7. Вне scope (анти-scope)

- ❌ Реальный торговый бот / исполнение ордеров.
- ❌ Собственный графический движок (берём lightweight-charts).
- ❌ Реконструкция стакана в реальном времени (L2). Только снапшоты — и то стретч.
- ❌ Поддержка нескольких бирж. Одна, сделанная нормально.
